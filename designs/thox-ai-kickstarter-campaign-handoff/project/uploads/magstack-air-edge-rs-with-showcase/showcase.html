What it is

A credible Air-class AI appliance slice.

The showcase is deliberately internal/prototype-safe: it presents working local AI jobs and cluster behavior without implying a tiny Pi node runs a large general-purpose LLM.

Control plane

Leader node with persistent queue

SQLite-backed task state, node registry, stale-job requeue, capability routing, authenticated API, and dashboard serving from the local network.

Node plane

Rust worker daemon

Telemetry, registration, beat loop, one-task default guardrail, signed internal flow, and a safe executor for built-in AI jobs or allowlisted plugins.

Experience layer

Dashboard + Python SDK

Static LAN dashboard for demos and a dependency-light Python SDK for product workflows, tests, and higher-level integrations.

Device anatomy

Low-profile compute node, stack-first identity.

The P0 goal is to prove physical stack behavior, Wi‑Fi cluster reliability, thermal behavior, and real local inference before advancing shared power or identity contacts.

Runtime fabric

Wi‑Fi replaces the backplane, Rust keeps it lean.

The realistic P0 design trades mDNS-only election for an explicit leader URL during hardware demos, while still using a zero-extra-cable Wi‑Fi fabric across stacked nodes.

Real edge AI jobs

Built-in tasks that can run on Air-class nodes.

These are not placeholders. They are small enough to be useful for prototype demos while preserving the upgrade path to ThoxMini, Nova, Jetson, or NPU-backed nodes.

ai.intent.v1

Tiny intent classifier

Bag-of-words softmax for routing prompts into operations, health, code, research, or general workflows without a cloud call.

ai.sensor_anomaly.v1

Telemetry anomaly detector

Z-score anomaly detection for thermal, load, battery drop, motion, or other compact numeric vectors.

ai.embed_hash.v1

Deterministic text vector

Normalized hashed embeddings for local routing, dedupe, cache keys, and approximate matching.

plugin.process.v1

Guarded plugin bridge

Optional allowlisted external process path for ONNX, TFLite, or llama.cpp wrappers on bigger Thox nodes.

fleet telemetry

AI-aware scheduling

Capabilities, temperature, memory, load, and stale task recovery give demos a realistic operational feel.

claim boundary

No large-LLM overclaim

Air runs lightweight AI and routes heavier jobs to stronger local nodes. That keeps the prototype honest and shippable.

Demo vertical slice

What the prototype should show on the bench.

This is the end-to-end story for a realistic working prototype: physical stack, Wi‑Fi cluster, local AI job, dashboard proof, and failure recovery.

Build
Assemble three Air nodes.

Use independently powered nodes for P0, validate printed enclosure fit, magnet polarity, antenna clearance, vents, service access, and thermal pad placement.

Install
Deploy Rust runtime.

Install magair-node, model JSON files, systemd unit, token config, and static dashboard on the leader.

Leader
Start the leader service.

Leader exposes authenticated API, dashboard, task queue, node registry, and result history on the local Wi‑Fi network.

Workers
Register and beat.

Each worker reports node ID, labels, capabilities, temperature, memory, load, and capacity.

Submit
Send an AI request.

Use dashboard, CLI, SDK, or local app bridge to submit ai.intent.v1, ai.sensor_anomaly.v1, or ai.embed_hash.v1.

Observe
Watch capability routing.

Dashboard shows queued, running, completed, worker assignment, and result preview.

Failure
Kill a worker or leader process.

Verify stale running tasks are requeued after timeout and completed by an available worker after restart.

Evidence
Export logs and screenshots.

Collect dashboard state, journal logs, task history, and thermal notes for the next hardware/software revision.
