# ADR 0005 - The UX engine is the single source of truth for design tokens

Date: 2026-06-06  |  Status: Accepted

## Context

The design system shipped two token surfaces that could drift apart: the flat
`design-systems/thox-ai-txf/TOKENS.json` consumed by the `txf-design-tokens`
Rust crate, and the brand rules that lived in prose and in a separate THOX brand
skill. Nothing reconciled them, and nothing consumed the token JSON to generate
and verify per-platform output. The branding and accessibility validators
self-attested via manifest booleans; their own source comments deferred the real
checks (token-bundle diff, contrast) to a future that had not arrived. This is
the same failure mode ADR-0004 names: a design system that is selectively
ignored because it is not mechanically enforced.

## Decision

Adopt the THOX UX engine (`tools/ux-engine`, package `@thox/txf-ux-engine`) as
the single source of truth for design tokens and the authoritative enforcer of
brand and accessibility rules.

1. Tokens are authored once in `tools/ux-engine/tokens/` (core primitives plus
   semantic theme overlays: base, meshstack, doc).
2. The engine compiles that source to every platform (CSS, Tailwind, SwiftUI,
   Compose, Rust, Figma) and to the TXF v2.0 runtime bundle.
3. The consumed `design-systems/thox-ai-txf/TOKENS.json` is generated by
   `node tools/ux-engine/src/sync-txf.mjs`. Brand colors derive from the engine
   core palette; TXF-specific structural scales are authored once in
   `tokens/targets/txf-v2.overrides.json`.
4. Four gates run in CI (job `ux-engine`): forbidden-pattern brand and
   compliance scan, WCAG 2.1 AA contrast across every theme, component-contract
   coverage, and `dist` drift. A fifth check, `sync-txf.mjs --check`, fails the
   build if `TOKENS.json` is hand-edited out of sync with the engine.

## Consequences

- There is exactly one place a token value is authored. `TOKENS.json` can no
  longer be edited out of band; CI rejects drift.
- The branding and accessibility validators keep their in-bundle scoring role,
  but the real enforcement (token diff, contrast math) now happens in the engine
  job on every PR. This realizes the future their source comments described.
- The Rust `txf-design-tokens` crate is unaffected: the generated bundle is the
  same flat v2.0 schema, value-preserving, so it deserializes unchanged.
- Adding a platform target is a new emitter in the engine, not a hand-maintained
  file per surface. Adding a brand rule is a new lint with a test and fixture.
- The TXF Experience Score gains a defensible branding and accessibility basis:
  the engine job is a required check upstream of `thox-cert`.
